## Context

В проекте уже есть карточка scoring-записи с редактируемыми секциями, отдельным блоком критериев выбора и сохранением данных через patch-модель записи. Предоценка сейчас присутствует как этап процесса, но не как структурированный блок с рисками, критичностью и итоговым решением.

Скриншот задает табличный формат блока из текущего рабочего файла: заголовок `Предоценка`, строки риска, колонка комментария, колонка критичности, ссылка на базу рисков и итоговые строки решения. В этом change проектируется web-аналог блока внутри scoring, без желтого Excel-оформления и без генерации Excel-файла.

## Goals / Non-Goals

**Goals:**

- Ввести отдельную модель `preassessment` в scoring-записи.
- Хранить строки риска отдельно от критериев выбора победителя.
- Поддержать ручное редактирование строк риска, критичности и итоговых полей.
- Отобразить блок в карточке записи в плотном табличном виде, визуально близком к Excel-блоку.
- Подготовить данные блока к будущему использованию в AI-pass и других внутренних интеграциях scoring.

**Non-Goals:**

- Не проектировать полноценный справочник/админку базы рисков в этом change.
- Не автоматизировать сметную оценку и расчет стоимости.
- Не генерировать Excel-файл и не обновлять Excel-export/coding-file generator.
- Не переносить логику блока в существующую модель `selectionCriteriaRows`.
- Не менять требования capability `selection-criteria-block`, кроме соседнего расположения на странице.

## Decisions

### 1. Отдельная модель `preassessment`

Решение: добавить в запись отдельный объект `preassessment`, а внутри него хранить `riskRows`, `summaryDecision`, `alexanderDecision`, `estimateFileUrl` и опциональную `riskBaseUrl`.

Причина: блок отвечает на вопрос “можем ли участвовать и как обходим риски”, а не “по каким критериям заказчик выбирает победителя”. Если смешать его с `selectionCriteriaRows`, появятся неверные статусы, названия полей и будущие ошибки в scoring input.

Альтернатива: хранить предоценку как обычные `notes`. Отказ: заметки не дают строк, статусов критичности и машинно-читаемых итогов.

### 2. Строки риска как упорядоченный список

Решение: `riskRows` хранится массивом строк с полями `order`, `parameter`, `managerComment`, `criticality`, `sourceKey` или `riskBaseRef`. В UI показывается фиксированная ссылка на внешнюю таблицу `База рисков` вместо подписи под заголовком; отдельный пользовательский input для ссылки не нужен. `riskBaseUrl` остается в модели только для совместимости и будущих интеграций, а будущая сервисная страница базы рисков проектируется отдельным change.

Причина: исходный блок табличный, порядок строк важен для восприятия web-аналога. Отдельная ссылка на источник риска позволит позже связать строку с базой рисков без жесткой зависимости от справочника.

Альтернатива: хранить риски как key-value объект. Отказ: объект хуже сохраняет порядок, сложнее поддерживает ручное добавление и дублирующиеся параметры.

### 3. Ограниченные статусы критичности

Решение: на первом проходе поддержать `critical`, `notCritical`, `unknown` с русскими label: `Критично`, `Не критично`, `Не указано`.

Причина: на скриншоте виден минимум `Не критично`, а колонка называется `Критично/Не критично`. `unknown` нужен, чтобы черновая строка могла существовать до принятия решения, но сохранение meaningful-строки должно подсвечивать отсутствие статуса.

Альтернатива: строковые произвольные значения. Отказ: это ломает фильтрацию, проверку и будущий экспорт.

### 4. Итоговые строки хранятся отдельно от рисков

Решение: `summaryDecision`, `alexanderDecision` и `estimateFileUrl` не входят в `riskRows`.

Причина: итоговые строки имеют другой смысл, другой набор контролов и не должны считаться рисками. Это также упростит поддержку web-аналога табличного блока.

Альтернатива: положить итоговые строки в общий массив с типом `summary`. Отказ: усложняет UI и повышает риск, что итоговая строка попадет в аналитику рисков.

### 5. Итоговое решение ограничено двумя значениями

Решение: `summaryDecision` и `alexanderDecision` поддерживают только `estimate` и `decline`, которые в интерфейсе отображаются как `Оценка` и `Не участвуем`.

Причина: в текущем процессе достаточно зафиксировать, идем ли в оценку или отказываемся от участия после предоценки. `Решение Александра` остается персональным label, но не должно быть произвольным текстом. Более широкий список сейчас добавит лишнюю неопределенность в приемку.

Альтернатива: добавить статусы вроде `На уточнении`, `Пауза`, `Нужны данные`. Отказ: это не подтверждено текущим рабочим сценарием.

### 6. UI ниже критериев выбора

Решение: показывать блок ниже секции `Критерии выбора`.

Причина: риски предоценки должны быть нижним блоком карточки, а `Критерии выбора` остаются ближе к тендерной информации.

Альтернатива: вынести блок в правый sidebar. Отказ: таблица рисков широкая, sidebar не подходит для комментариев и плотной табличной структуры.

## Risks / Trade-offs

- [Risk] Полный список параметров находится во внешней таблице базы рисков. → Mitigation: сделать ручные строки и внешнюю ссылку на таблицу; сервисную страницу базы рисков оставить отдельным будущим change.
- [Risk] Табличный блок будет выглядеть как чужеродная Excel-вставка. → Mitigation: использовать текущие цвета, radius, borders и form controls проекта, без желтого оформления.
- [Risk] `Решение Александра` является персонализированным полем и может измениться организационно. → Mitigation: оставить label и техническое поле `alexanderDecision` по текущему пользовательскому решению; переименование делать только отдельным change.
- [Risk] AI-pass начнет писать частичные данные без критичности. → Mitigation: нормализация должна принимать черновые строки, но UI и проверки должны явно показывать незаполненную критичность.
- [Risk] Блок увеличит высоту страницы. → Mitigation: использовать компактную таблицу/строки, не карточки внутри карточек, и адаптивный вид на мобильных.

## Migration Plan

1. Добавить нормализацию `preassessment` с безопасным default: пустой объект, пустые строки, пустые итоговые поля.
2. Расширить patch/save payload так, чтобы существующие записи без `preassessment` продолжали открываться.
3. Добавить UI блока и сохранение в карточке записи.
4. При необходимости добавить демо-данные или шаблон строк риска только для новых/пустых записей.
5. Добавить smoke/проверку нормализации и сохранения.

Rollback: удалить UI-блок из карточки и перестать передавать `preassessment` в patch payload. Существующие сохраненные данные в JSON можно оставить как неиспользуемое поле, если они не мешают нормализации.
