## ADDED Requirements

### Requirement: Блок предоценки рисков
Система SHALL предоставлять на scoring-записи отдельный блок `Предоценка (заполняет - сметчик)` для фиксации рисков участия и решений по предоценке.

#### Scenario: Пользователь открывает scoring-запись
- **WHEN** пользователь открывает страницу scoring-записи
- **THEN** система SHALL показывать блок предоценки как отдельный раздел, не смешивая его с блоком критериев выбора

#### Scenario: Пользователь читает тендерную информацию
- **WHEN** пользователь просматривает секцию `Информация по тендеру`
- **THEN** система SHALL располагать блок предоценки ниже блока `Критерии выбора`

### Requirement: Табличные строки риска
Система SHALL представлять каждый риск предоценки отдельной строкой с параметром, комментарием и критичностью.

#### Scenario: В блоке есть риск
- **WHEN** риск добавлен в блок предоценки
- **THEN** система SHALL хранить и показывать для него `Параметр`, `Комментарий менеджера и/или предложение о том можем ли обойти данный риск` и `Критично/Не критично`

#### Scenario: Пользователь добавляет несколько рисков
- **WHEN** пользователь добавляет несколько строк риска
- **THEN** система SHALL сохранять и показывать строки в порядке добавления

### Requirement: Ручное управление строками риска
Система SHALL позволять пользователю вручную добавлять, редактировать и удалять строки риска предоценки.

#### Scenario: Пользователь добавляет строку риска
- **WHEN** пользователь добавляет новую строку в блок предоценки
- **THEN** система SHALL создать отдельную строку риска с новым порядковым номером

#### Scenario: Пользователь удаляет строку риска
- **WHEN** пользователь удаляет строку риска
- **THEN** система SHALL убрать строку из блока и пересчитать порядок оставшихся строк

### Requirement: Ограниченный статус критичности
Система SHALL поддерживать ограниченный набор статусов критичности для строк риска: `Критично`, `Не критично`, `Не указано`.

#### Scenario: Пользователь выбирает критичность
- **WHEN** пользователь выбирает значение в колонке `Критично/Не критично`
- **THEN** система SHALL сохранить нормализованный статус критичности для строки риска

#### Scenario: Строка риска еще не оценена
- **WHEN** строка риска имеет параметр или комментарий, но критичность не выбрана
- **THEN** система SHALL показывать строку как незавершенную, не подставляя `Критично` или `Не критично` автоматически

### Requirement: Внешняя ссылка на базу рисков
Система SHALL показывать в блоке предоценки фиксированную внешнюю ссылку на таблицу `База рисков`.

#### Scenario: Пользователь открывает блок предоценки
- **WHEN** пользователь открывает блок предоценки
- **THEN** система SHALL показывать ссылку `База рисков` в шапке блока вместо подписи `Заполняет сметчик`

#### Scenario: Пользователь работает со строками риска
- **WHEN** пользователь редактирует строки риска
- **THEN** система SHALL NOT требовать отдельный input для ссылки на базу рисков

#### Scenario: Сервисная страница базы рисков появится позже
- **WHEN** в будущем появится сервисная страница базы рисков
- **THEN** система SHALL позволять заменить внешнюю ссылку без изменения модели строк риска

### Requirement: Визуальное соответствие UI проекта
Система SHALL отображать блок предоценки в стиле текущего UI проекта, а не в желтом Excel-оформлении.

#### Scenario: Пользователь просматривает блок предоценки
- **WHEN** пользователь видит блок предоценки в web-карточке
- **THEN** система SHALL использовать общие цвета, границы, скругления и form controls проекта

#### Scenario: Блок сравнивается с исходной таблицей
- **WHEN** web-блок повторяет структуру исходной таблицы
- **THEN** система SHALL сохранять смысловую структуру строк и итогов без копирования желтого визуального оформления

### Requirement: Итоговые решения предоценки
Система SHALL хранить итоговые поля предоценки отдельно от строк риска.

#### Scenario: Пользователь фиксирует итог предоценки
- **WHEN** пользователь заполняет `Решение по итогам предоценки`
- **THEN** система SHALL сохранить это значение как итоговое решение блока, а не как строку риска

#### Scenario: Пользователь фиксирует решение Александра
- **WHEN** пользователь выбирает `Решение Александра`
- **THEN** система SHALL сохранить это значение как отдельное поле `Решение Александра`, а не как строку риска

#### Scenario: Пользователь указывает файл оценки
- **WHEN** пользователь заполняет `Файл оценки`
- **THEN** система SHALL сохранить ссылку или идентификатор файла оценки отдельно от строк риска

### Requirement: Значения итоговых решений
Система SHALL поддерживать только значения `Оценка` и `Не участвуем` для `Решение по итогам предоценки` и `Решение Александра`.

#### Scenario: Пользователь выбирает оценку
- **WHEN** пользователь выбирает значение `Оценка`
- **THEN** система SHALL сохранить это значение как нормализованное итоговое решение предоценки

#### Scenario: Пользователь выбирает отказ от участия
- **WHEN** пользователь выбирает значение `Не участвуем`
- **THEN** система SHALL сохранить это значение как нормализованное итоговое решение предоценки

#### Scenario: Итоговое решение не выбрано
- **WHEN** пользователь еще не выбрал итоговое решение
- **THEN** система SHALL хранить пустое значение и не считать блок завершенным по итогам предоценки

#### Scenario: Пользователь фиксирует решение Александра из списка
- **WHEN** пользователь выбирает `Решение Александра`
- **THEN** система SHALL использовать тот же выпадающий список `Оценка` / `Не участвуем`

### Requirement: Сохранение блока в scoring-записи
Система SHALL сохранять блок предоценки в составе scoring-записи через read/update контракт записи.

#### Scenario: Пользователь сохраняет карточку
- **WHEN** пользователь сохраняет scoring-запись с заполненным блоком предоценки
- **THEN** система SHALL записать строки риска, ссылку на базу рисков, итоговые решения и файл оценки в модель записи

#### Scenario: Пользователь заново открывает карточку
- **WHEN** пользователь повторно открывает scoring-запись после сохранения
- **THEN** система SHALL восстановить блок предоценки из сохраненной модели записи

### Requirement: Совместимость с существующими записями
Система SHALL открывать существующие scoring-записи без блока предоценки без ошибки.

#### Scenario: В записи нет данных предоценки
- **WHEN** пользователь открывает старую scoring-запись без поля предоценки
- **THEN** система SHALL показать пустой блок предоценки или состояние отсутствия строк

#### Scenario: Пользователь сохраняет старую запись
- **WHEN** пользователь сохраняет старую scoring-запись после появления нового блока
- **THEN** система SHALL не терять существующие поля записи и SHALL сохранить предоценку в новой модели, если пользователь ее заполнил

### Requirement: Web-аналог табличного блока и AI-pass
Система SHALL держать данные предоценки в машинно-читаемой форме, пригодной для web-аналога табличного блока и будущего AI-pass.

#### Scenario: Runtime читает scoring-запись
- **WHEN** runtime или AI-pass читает scoring-запись
- **THEN** система SHALL предоставлять блок предоценки как структурированный объект, а не как один текстовый комментарий

#### Scenario: Web-карточка отображает аналог таблицы
- **WHEN** web-карточка получает scoring-запись с блоком предоценки
- **THEN** система SHALL отображать табличный web-аналог блока без генерации Excel-файла
