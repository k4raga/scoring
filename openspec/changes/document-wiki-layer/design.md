## Context

В карточке проекта уже есть правый блок "Документы и ссылки", `record.documents`, `record.documentArtifacts`, страница Markdown-просмотра и extractor-артефакты: original files, `normalized/*.md`, `knowledge/*.html`, diagnostic JSON. Сейчас эти сущности связаны слабо: UI показывает группы артефактов, но у системы нет явной пользовательской модели "wiki/MD-документации проекта".

Новый слой нужен для разделения трех смыслов:

- оригиналы документов как источник истины;
- wiki/MD-документы как удобная производная и нормализованная документация проекта;
- технические JSON/manifest/report как диагностика extractor-run.

Дополнительный целевой контекст из обсуждения: wiki/MD-слой должен быть совместим с отдельной Quartz-базой знаний или аналогичным static wiki renderer. В этой базе должны жить проекты и все связанные MD-документы, чтобы их можно было просматривать не только из карточки scoring, но и как самостоятельную knowledge base.

## Goals / Non-Goals

**Goals:**

- Ввести отдельную модель wiki/MD-блоков проекта.
- Автоматически собирать правый блок "Документы и ссылки" из оригиналов, wiki/MD-документов и технических артефактов.
- Дать ручное управление блоками: добавить, скрыть, удалить, переименовать, изменить порядок.
- Заложить совместимость с будущим Quartz/wiki-контуром, где проектные MD-документы публикуются как база знаний.
- Определить normalization pass для MD: документ должен быть структурирован заголовками, списками, таблицами и логическими секциями, насколько это возможно без бизнес-анализа.
- Сохранить совместимость со старыми `documents` и `documentArtifacts`.
- Оставить extractor отдельным сервисом, который не принимает бизнес-решений.

**Non-Goals:**

- Не строить полноценную LLM-аналитику закупки.
- Не делать полноценную настройку и деплой Quartz внутри текущего implementation pass, если это не будет выделено отдельной задачей.
- Не конвертировать все исходные документы в идеальные MD-страницы любой ценой.
- Не менять основной деплойный контур и storage-стратегию.

## Decisions

### Decision 1: Wiki/MD layer is project metadata, not extractor output

В `record` должен появиться отдельный слой конфигурации wiki-документации, например `documentWiki` или близкая по смыслу структура. Он хранит пользовательский порядок, видимость, подписи и ручные блоки.

Extractor продолжает отдавать технические артефакты. Scoring backend нормализует их в пользовательскую модель правого блока.

Alternative considered: использовать `documents` как единственный источник. Это проще, но снова смешивает оригиналы, MD, JSON и ручные блоки.

### Decision 2: Right-side document block is generated from a view model

Frontend не должен самостоятельно угадывать смысл каждого artifact. Backend должен отдавать готовую или почти готовую view model:

- source/original documents;
- wiki/MD documents;
- manual blocks;
- diagnostic artifacts.

Frontend отвечает за отображение и ручное редактирование, а не за бизнес-классификацию источников.

Alternative considered: оставить текущую классификацию в `DetailPage.jsx`. Это быстрее, но усилит связанность UI с extractor-форматами.

### Decision 3: Manual edits override generated defaults

Автогенерация должна быть повторяемой, но не должна стирать пользовательские правки. Если пользователь переименовал блок, скрыл его или поменял порядок, эти настройки имеют приоритет над новым generated-state.

Для этого нужен стабильный идентификатор блока:

- archive/source file id;
- wiki document id;
- artifact key;
- manual block id.

### Decision 4: Original files remain first-class links

Оригинальные файлы должны открываться по прямой ссылке или скачиваться браузером. MD/wiki-слой не заменяет оригинал и не должен быть единственным способом доступа к документу.

### Decision 5: Diagnostic artifacts are visually secondary

JSON, manifest, extraction report и похожие файлы должны быть доступны, но в отдельной технической группе. В обычном пользовательском сценарии они не должны выглядеть как основные документы.

### Decision 6: Quartz/wiki compatibility is a first-class target

Wiki/MD-документы должны храниться и описываться так, чтобы их можно было публиковать в отдельный Quartz/wiki-контур без переписывания модели. Для этого у проекта и MD-документов должны быть стабильные id, человекочитаемые title, связи project-to-document и предсказуемые пути.

На текущем этапе scoring может отображать wiki/MD-слой внутри своей карточки, но модель не должна быть закрыта только на embedded viewer. Она должна позволять собрать отдельную базу знаний по проектам и MD-документам.

Alternative considered: встроить всю wiki-логику только в React UI scoring. Это быстрее, но потом усложнит перенос в Quartz и сделает knowledge base зависимой от основного приложения.

### Decision 7: Markdown normalization is structural, not analytical

При преобразовании документа в MD extractor или следующий normalization step должен делать документ человекочитаемым: восстанавливать заголовки, списки, таблицы, разделы, абзацы и базовую иерархию. Это техническая нормализация документации, а не принятие решений по закупке.

Нормализация может использовать эвристики и разбор форматов (`docx`, `xlsx`, `pdf`, text), но не должна определять победителей, риски, итоговые scoring-поля или бизнес-смысл сверх структуры исходного документа.

Alternative considered: сохранять MD как прямой text dump. Это дешевле, но плохо подходит для wiki/knowledge-base слоя и заставляет пользователя читать нечитаемые документы.

## Risks / Trade-offs

- [Risk] Старые записи содержат неоднородные `documents` и локальные artifact-ссылки → Mitigation: добавить compatibility mapper и не требовать миграции всех записей сразу.
- [Risk] Ручные правки могут конфликтовать с повторной автогенерацией → Mitigation: хранить manual overrides отдельно от generated source.
- [Risk] UI редактирования блоков может разрастись → Mitigation: начать с минимального управления: title, type, href/body, visible, order.
- [Risk] Пользователь может удалить автоматически созданный блок и потом потерять ссылку → Mitigation: реализовать hide/restore для generated blocks, а hard delete оставить только для manual blocks.

## Migration Plan

1. Добавить backend view model для блока документов на основе текущих `documents` и `documentArtifacts`.
2. Добавить новую структуру пользовательских overrides без удаления старых полей.
3. Обновить UI правого блока на новую view model.
4. Добавить ручное управление блоками.
5. Уточнить и реализовать правила технической нормализации MD-документов.
6. Зафиксировать mapping проекта и MD-документов для будущей Quartz/wiki-публикации.
7. Обновить документацию по модели документов, wiki/MD-слою и knowledge-base контурy.

Rollback: оставить старые `documents` и `documentArtifacts` неизменными; при ошибке UI может временно вернуться к текущему mapper.
