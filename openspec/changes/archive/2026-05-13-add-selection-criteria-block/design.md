## Статус

Черновик технического подхода после первичного интервью.

Целевая продуктовая модель уже задана пользователем. Текущая скрытая реализация критериев должна быть удалена или заменена; она не является основой design.

## Контекст

Блок критериев выбора нужен менеджеру по тендерам для анализа документации закупки. В документах описываются правила выбора победителя: цена, неценовые преимущества и дополнительные требования. Для scoring важно не просто хранить ссылку на документ, а видеть список критериев и понимать, закрываем ли мы каждый пункт.

В коде есть старая скрытая реализация, но она признана некорректной. Поэтому implementation должен начинаться с инвентаризации и удаления/замены старого поведения.

## Цели / не цели

**Цели:**

- Реализовать новый блок критериев выбора внизу страницы под "Информация по тендеру".
- Сделать одну строку одним критерием или одним дополнительным требованием.
- Поддержать группировку и порядок строк.
- Поддержать типы строк: ценовой критерий, неценовой критерий, дополнительное требование.
- Поддержать вес для ценовых и неценовых критериев.
- Сделать статус закрытия обязательным полем.
- Дать пользователю возможность добавлять и удалять строки визуально.
- Подготовить модель к будущему заполнению через внешнюю/AI extraction-интеграцию.
- Включить тестовое заполнение через extractor в первый implementation pass.
- Использовать порядок добавления строк как порядок отображения на первом этапе.
- Хранить вес ценовых и неценовых критериев как процент.
- Удалить или заменить старую скрытую реализацию.
- Не переносить старые criteria-данные в новую модель.

**Не цели:**

- Не дорабатывать старую скрытую реализацию как базу.
- Не рассчитывать победителя тендера в рамках этой задачи.
- Не реализовывать полноценную production-интеграцию внешнего extractor/AI-контура, если она не выделена отдельной задачей.
- Не менять всю страницу карточки проекта вне размещения нового блока.

## Решения

### Решение 1: новая модель строки вместо старой criteria-модели

Целевая строка должна описывать один критерий или одно дополнительное требование. Минимальные поля:

- `order` — порядок;
- `group` — ценовой, неценовой, дополнительное требование;
- `title` — название критерия;
- `weightPercent` — вес в процентах для ценовых и неценовых критериев, если указан;
- `coverageStatus` — обязательный статус: полностью закрываем, частично закрываем, не закрываем;
- `coverageNote` — как именно закрываем или почему не закрываем;
- `source` или `sourceExcerpt` — техническая ссылка/выдержка из документации, если доступна из extractor/API; отдельный пользовательский блок "Источник / выдержка" в первом UI не нужен.

Почему так: такая модель отвечает пользовательскому сценарию и может стать структурированным scoring-входом.

### Решение 2: блок размещается под "Информация по тендеру"

UI должен показывать блок внизу страницы после блока "Информация по тендеру". Это делает критерии частью тендерного анализа, а не техническим приложением к документам.

Почему так: менеджер читает критерии в контексте условий тендера.

### Решение 3: группировка и порядок обязательны, ручная сортировка не нужна в первом этапе

Критерии должны иметь порядок и группировку по типам. В первом этапе порядок определяется порядком добавления строк; отдельная ручная сортировка не нужна.

Почему так: плоский список хуже отражает реальную структуру критериев в тендерной документации.

### Решение 4: статус закрытия обязателен

Каждая строка должна явно отвечать, закрываем ли мы критерий или требование полностью, частично или не закрываем.

Почему так: для scoring важен не только факт наличия критерия, но и наша позиция по нему.

### Решение 5: старая реализация удаляется или заменяется

Нельзя добавить новый блок поверх скрытого старого поведения. Implementation должен удалить старую пользовательскую поверхность и заменить старый data-flow новым контрактом. Старые данные критериев не переносятся в новую модель, потому что они не считаются корректными.

Почему так: две модели критериев создадут расхождение между UI, backend и scoring input.

### Решение 6: `AI-pass` является будущим источником заполнения, тестовый extractor обязателен для первого pass

В будущем блок должен заполнять `AI-pass`. В первом implementation pass нужен тестовый extractor, который пишет данные в новую модель и не закрепляет старую некорректную структуру.

Почему так: это позволяет проверить сценарий без ожидания полноценной интеграции.

### Решение 7: вес хранится как процент

Вес ценовых и неценовых критериев должен храниться как процент. Другие форматы на этом этапе не поддерживаются.

Почему так: в текущем доменном сценарии пользователь не встречал других форматов веса.

## Инвентаризация старой реализации

Найденные legacy-места:

- Frontend: `frontend/src/pages/DetailPage.jsx` содержал состояние и handlers для старых `criteriaRows`, но пользовательский блок не был целевой реализацией.
- Backend schema/API: `backend/src/record-schema.js`, `backend/src/record-patch.js`, `backend/src/data-store.js`, `backend/src/record-ingest.js` читали и нормализовали старые `criteriaRows` / `criteria`.
- Export/runtime: `backend/src/xlsx-export.js` и `buildExcelSections` использовали старые группы как источник вывода.
- Extractor: `artifacts/scoring-extractor/server.py` писал старую структуру `criteriaRows`.
- Данные: `backend/data/coding-records.json` уже содержит старые `criteriaRows`; эти данные не переносятся в новую модель.

Действия:

- Новый пользовательский и scoring-контракт: только `selectionCriteriaRows`.
- Старые `criteriaRows` / `criteria` не передаются в новый UI-блок и не являются scoring input.
- Legacy-поля допускаются только как временная compatibility/diagnostic поверхность в сохраненных данных и patch-контракте.
- Extractor первого pass пишет в `selectionCriteriaRows`, а не в `criteriaRows`.
- Export и excel-секции читают новую модель.

## Риски / компромиссы

- [Риск] Тестовый extractor закрепит временный формат -> Митигация: extractor должен писать в новую модель строки.
- [Риск] Старые данные несовместимы с новой моделью -> Митигация: не переносить старые данные; при необходимости оставить только диагностическое чтение на время удаления старой реализации.
- [Риск] Вес отсутствует в документации -> Митигация: вес в процентах поддерживается, но обязательность веса зависит от группы и наличия данных в документации; обязательным является статус закрытия.
- [Риск] Пользовательский блок станет слишком тяжелым -> Митигация: группировка, компактные строки и редактирование по одной строке.

## План миграции

1. Инвентаризировать старую скрытую criteria-реализацию.
2. Удалить старый UI или исключить его из пользовательского сценария.
3. Не переносить старые criteria-данные в новую модель.
4. Ввести новую модель строк критериев.
5. Реализовать новый блок под "Информация по тендеру".
6. Подключить ручное добавление/удаление строк.
7. Подключить тестовый extractor к новой модели в первом implementation pass.
8. Проверить, что scoring/export/API не читают старую некорректную модель как источник истины.

Rollback: если новая модель не принимается, откат должен вернуть страницу без блока критериев, но не должен снова делать старую скрытую реализацию источником истины.

## Открытые вопросы

- Нет открытых продуктовых вопросов по первому pass. Технические детали implementation уточняются на этапе инвентаризации старой реализации.
