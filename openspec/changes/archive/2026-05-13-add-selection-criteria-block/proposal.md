## Статус

Черновик после первичного интервью с пользователем.

Согласовано: блок критериев выбора проектируется заново; текущая скрытая реализация должна быть удалена или заменена и не считается корректной основой.

Согласованы оставшиеся продуктовые уточнения: будущий источник заполнения называется `AI-pass`, тестовый extractor нужен в первом implementation pass, на первом этапе достаточно порядка добавления строк, вес критерия хранится как процент, старые данные переносить не нужно.

## Пользовательский сценарий

Менеджер по тендерам изучает документацию закупки и ищет критерии, по которым организатор тендера будет выбирать победителя среди участников.

В документации обычно описано, почему один участник побеждает другого: например, из-за наименьшей цены, набора неценовых преимуществ или выполнения дополнительных требований. Менеджеру нужно видеть эти критерии в карточке scoring-записи, понимать их вес, порядок и то, закрываем ли мы каждый критерий полностью, частично или не закрываем.

## Зачем

Нужен новый явный блок критериев выбора, чтобы критерии из документации стали рабочей частью scoring-контура, а не скрытой или случайной технической структурой.

Блок должен помогать оценивать, насколько наше предложение соответствует правилам выбора победителя, и давать структурированные данные для дальнейшей автоматизации scoring.

## Что меняется

- Проектируется и реализуется новый блок критериев выбора.
- Блок отображается внизу страницы, под блоком "Информация по тендеру".
- Одна строка блока соответствует одному критерию или одному дополнительному требованию.
- Критерии имеют порядок и группировку.
- Поддерживаются ценовые критерии, неценовые критерии и дополнительные требования.
- Для ценовых и неценовых критериев поддерживается вес.
- Для каждой строки обязательно фиксируется, закрываем ли мы критерий/требование полностью, частично или не закрываем.
- Пользователь должен иметь возможность визуально добавлять и удалять строки критериев.
- В будущем блок должен заполнять `AI-pass`; на первом этапе допустим тестовый extractor, если он пишет данные в целевую модель.
- Тестовый extractor входит в первый implementation pass.
- На первом этапе отдельная ручная сортировка не требуется: порядок определяется порядком добавления строк.
- Вес ценовых и неценовых критериев задается в процентах.
- Текущую скрытую реализацию критериев нужно удалить или заменить, а не достраивать поверх нее.
- Старые данные критериев не переносить в новую модель, потому что они не считаются корректными.

## Целевая модель строки

Минимальная строка критерия должна поддерживать:

- порядок внутри блока;
- группу: ценовой критерий, неценовой критерий, дополнительное требование;
- название критерия или требования;
- вес для ценовых и неценовых критериев, если он указан в документации;
- формат веса: процент;
- обязательный статус закрытия: полностью закрываем, частично закрываем, не закрываем;
- пояснение, каким образом мы закрываем или не закрываем критерий;
- технический источник или выдержку из документации, если она доступна из extractor/API; отдельное пользовательское поле источника в первом UI не требуется.

## Capabilities

### New Capabilities

- `selection-criteria-block`: новый явный блок критериев выбора scoring-записи, включая целевой UX, модель строки, ручное редактирование, будущий `AI-pass` и пригодность для scoring-входа.

### Modified Capabilities

- Нет. В `openspec/specs/` пока нет архивированных capabilities, которые можно модифицировать через delta.

## Влияние

- Product/UX:
  - новый пользовательский сценарий для менеджера по тендерам;
  - блок внизу страницы под "Информация по тендеру";
  - ручное добавление и удаление строк.
- Frontend:
  - новый UI-блок критериев;
  - группировка, порядок, вес, статус закрытия и пояснения;
  - удаление текущей скрытой реализации из пользовательского сценария.
- Backend:
  - новая или пересмотренная модель хранения критериев;
  - read/update API для строк критериев;
  - удаление, замена или миграция старой некорректной структуры.
- Extractor/runtime:
  - тестовый extractor-pass в первом implementation pass;
  - будущий контракт заполнения блока через `AI-pass`.
- Workflow:
  - `TASK-1` связан с этим OpenSpec change;
  - implementation начинается только после обновления `design.md`, `spec.md` и `tasks.md` под согласованный сценарий.
