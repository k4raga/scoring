# Scoring Architecture

## Purpose

`scoring/` — отдельный контур для проектирования и реализации системы скорринга.

На старте проект должен удерживать три слоя:

- domain layer — сущности, правила, сценарии скорринга;
- product layer — пользовательские потоки, требования, ограничения;
- implementation layer — код, интеграции, инфраструктурные решения.

## Principles

- не смешивать внешние runtime-контуры с логикой `scoring`;
- не хранить черновую предоценку как основной source of truth, если проект уже выделен в отдельный контур;
- развивать проект как самостоятельный продукт с собственной документацией и кодом.

## Development contour

Помимо продуктовой архитектуры, в проекте теперь отдельно зафиксирован development contour.

Его задача:

- описывать роли участников разработки;
- фиксировать права основной консоли, developer и QA;
- задавать board/scenario loop для следующих циклов;
- не смешивать устройство команды с устройством продукта.

Ключевые артефакты:

- `AGENTS.md`
- `AGENT-LOOP.md`
- `.codex-workflow/team-policy.md`
- `.codex-workflow/task-board.md`
- `.codex-workflow/test-scenarios.md`

Это не замена архитектуре решения, а надстройка над ней: слой проектной разработки и приемки.

## Current business process

На текущем этапе в проекте уже есть один явно выделенный операционный процесс: `coding`.

Его роль:

- принять пакет документов по закупке;
- проанализировать материалы;
- заполнить таблицу кодинга;
- создать задачу в `Bitrix24`;
- зафиксировать результат как рабочий файл и связанную задачу.

Подробное описание процесса вынесено в `docs/CODING-PROCESS.md`.

## Current solution direction

Для процесса `coding` отдельно рассматривается архитектура AI-first решения.

Основной фокус:

- orchestration процесса;
- извлечение фактов из тендерных документов;
- генерация файла кодинга;
- создание задачи в `Bitrix24`;
- разделение control plane и execution plane.

Варианты решения и рекомендуемое направление описаны в `docs/CODING-SOLUTION-ARCHITECTURES.md`.

## Codex-first MVP

В качестве ближайшего практического шага в проект добавлен отдельный `Codex-first MVP` контур.

Его смысл:

- локально запускать `coding` как набор воспроизводимых `run`;
- хранить канонические артефакты процесса;
- использовать `Codex` как основной execution runtime;
- не смешивать MVP-контур с будущим отдельным сервисом.

Подробное описание вынесено в `docs/CODEX-FIRST-MVP.md`.

## Coding web direction

Отдельно фиксируется целевая архитектура будущего внутреннего сайта кодинга.

Ключевая идея:

- основной интерфейс работы должен быть web-based;
- навигация должна строиться по структуре `год -> месяц -> проект`;
- при загрузке архива должна автоматически создаваться страница проекта;
- после создания страницы должен запускаться анализ документов;
- `Codex` на первом этапе остается execution-слоем;
- позже orchestration можно вынести в `Dify`.

Подробная фиксация этой идеи вынесена в `docs/CODING-WEB-ARCHITECTURE.md`.
