# UI Reference

## Статус

Утвержденный дизайн-ориентир для текущего этапа проекта `scoring`.

Текущий каноничный комплект состоит из трех экранов:

- [prototype-home.html](/C:/Users/illki/Desktop/projects/scoring/prototype-home.html) — главный экран / реестр;
- [prototype-month.html](/C:/Users/illki/Desktop/projects/scoring/prototype-month.html) — месячный слой / архивный срез;
- [prototype-detail.html](/C:/Users/illki/Desktop/projects/scoring/prototype-detail.html) — детальная карточка проекта.

Именно этот набор фиксируется как baseline для следующих экранов до отдельного согласованного redesign-pass.

## Три Каноничных Дизайна

### 1. Home

- роль: основной вход в контур scoring;
- задача: показать список и навигацию по проектам;
- источник решений: шапка, общий page rhythm, карточки, фильтры, footer.

### 2. Month

- роль: промежуточный слой между реестром и деталкой;
- задача: держать месячную агрегацию, компактную статистику и переход в конкретные проекты;
- источник решений: месячные карточки, логика архива, компактные статусные блоки.

### 3. Detail

- роль: рабочая карточка одного проекта;
- задача: редактирование полей, просмотр сумм, параметров тендера и сопроводительных ссылок;
- источник решений: form cards, side rail, doc-link pattern, календарные поля и детальный layout.

## Что считается зафиксированным

Ниже перечислены не случайные детали одной страницы, а утвержденные паттерны, которые должны переиспользоваться на следующих страницах проекта.

### 1. Шапка

- Белая шапка на светлом фоне страницы.
- Слева знак `Scoring` и подпись-слоган:
  - `Утром — деньги, вечером — стулья`
- Справа компактная иконка поиска и акцентная кнопка действия.
- Общий характер:
  - чистая поверхность;
  - мягкая рамка;
  - без стеклянного эффекта;
  - без тяжелых декоративных приемов.

### 2. Главная типографика

- Крупный жирный заголовок секции.
- Геометричная гротескная подача.
- Темный текст без лишних цветовых акцентов в крупных заголовках.
- Вторичный текст спокойный, приглушенный, но читаемый.

### 3. Карточки проектов

- Светлые карточки с мягкой рамкой и аккуратной тенью.
- В верхней зоне карточки:
  - статус;
  - дата.
- Ниже:
  - короткое имя проекта;
  - краткое описание;
  - заказчик.
- Эти карточки считаются базовым паттерном для списков и лент проектов.

### 4. Фильтры по стадиям

- Фильтры стоят под заголовком секции, а не вплотную к нему.
- Используются pill/chip-контролы.
- Активный фильтр выделяется мягко, без агрессивного контраста.
- Сетка карточек не должна ломаться от фильтрации:
  - всегда держится фиксированное число слотов;
  - если данных не хватает, используются плейсхолдеры.

### 5. Архив по годам и месяцам

- Год — крупный текстовый маркер секции.
- Внутри года — сетка компактных карточек месяцев.
- Карточка месяца содержит:
  - название месяца;
  - счетчик проектов рядом с названием;
  - компактную разбивку по стадиям.
- Не использовать:
  - дубли названия месяца;
  - лишние CTA внутри карточки месяца;
  - второстепенные описательные подписи ради заполнения пространства.

### 6. Компактная стадийная статистика месяца

- Для месяца используется компактный формат.
- Не вертикальный длинный список.
- Предпочтительный формат:
  - короткие inline-бейджи;
  - цветовая привязка к статусам;
  - компактные числа без перегруза.

### 7. Подвал

- Небольшой спокойный footer внизу страницы.
- Без перегрузки навигацией.
- Используется как общий завершающий паттерн для страниц продукта.

## Библиотека паттернов, которую переносим дальше

При проектировании следующих страниц нужно считать утвержденными и повторно использовать:

- шапку;
- footer;
- проектные карточки;
- карточки месяцев;
- фильтры-чипы;
- статусные чипы;
- общую плотность воздуха, рамок и теней;
- светлую поверхностную архитектуру страницы.

## Что не надо делать заново

Если новая страница ложится в уже утвержденный язык, не нужно:

- заново переизобретать шапку;
- заново спорить о стиле карточек;
- заново собирать footer;
- менять визуальный ритм секций без отдельного решения;
- возвращаться к glassmorphism или к декоративным двухколоночным композициям без явной причины.

## Правило дальнейших изменений

Следующие страницы должны проектироваться относительно этого референса.

Изменение самого референса допустимо только если:

- переработка охватывает уже не одну страницу, а несколько;
- есть отдельное согласование, что меняется именно общий дизайн-язык;
- после этого обновляется этот документ и сам эталонный файл.
