# Codex-First MVP Runtime

## Что это

Папка `mvp/` содержит минимальный локальный runtime для процесса `coding`.

Это стартовый контур, в котором:

- каждый запуск оформляется как отдельный `run`;
- документы копируются в рабочую папку запуска;
- создаются базовые артефакты процесса;
- `Codex` потом может доработать этот запуск до готового кодинга.

## Базовая команда

```bash
python mvp/scripts/init_run.py --slug cemros-block --template tmp/sample-coding.xlsx path/to/doc1 path/to/doc2
```

## Что делает `init_run.py`

- создает новую папку запуска в `mvp/runs/`;
- создает подпапки `input/`, `normalized/`, `output/`;
- копирует входные документы в `input/`;
- если передан шаблон, копирует его как `output/coding.xlsx`;
- создает `facts.json`, `bitrix-task.json`, `run-log.json`, `summary.md`.

## Как дальше использовать run

После инициализации `run` можно отдавать `Codex` как рабочий контур.

Практически это выглядит так:

1. `Codex` читает документы из `input/`.
2. При необходимости делает текстовые выгрузки в `normalized/`.
3. Заполняет `facts.json`.
4. На основе фактов готовит `output/coding.xlsx`.
5. Подготавливает или создает задачу в `Bitrix24`.
6. Обновляет `bitrix-task.json` и `run-log.json`.

## Обязательное правило

Сначала должен появиться нормализованный слой фактов:

`input -> facts.json`

И только потом:

`facts.json -> coding.xlsx / Bitrix24`

Именно это защищает MVP от хаотичной логики "сразу писать Excel из документов".
