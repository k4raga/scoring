# Future Scope

Этот файл хранит согласованные пользовательские истории, которые еще не запущены в реализацию.

Текущее правило:
- baseline-версия приложения не меняется автоматически;
- каждая история сначала фиксируется здесь как scope, acceptance и functional QA;
- в активный цикл задача переносится только после явного одобрения пользователя.

## FS-001

### Title

Create-project ingest flow from home

### User Story

Пользователь заходит на главную страницу, нажимает `Добавить проект`, вводит название проекта, выбирает архив и создает проект.

После этого:
- создается новая запись;
- из архива подгружаются данные;
- новая запись стартует в стадии `Скоринг`;
- поля, по которым данные не пришли или не были распознаны, отображаются как выключенные, а не как обычные пустые редактируемые поля.

### Scope

1. Привести create-project flow на главной к продуктовой логике ingest, а не только к созданию пустой записи.
2. Явно разделить в detail два состояния:
   - данные пришли из ingest;
   - данные отсутствуют и поле пока выключено.
3. Зафиксировать стартовую стадию новой записи как `Скоринг`.
4. Показать пользователю, что архив был принят и разбор данных выполнен или выполняется.
5. Не менять канонический дизайн и не пересобирать существующую baseline-версию вне границ этой истории.

### Acceptance

- На `/` кнопка `Добавить проект` открывает popup создания проекта.
- Popup требует минимум:
  - название проекта;
  - архив проекта.
- После подтверждения backend создает новую запись и открывается detail-route новой записи.
- Новая запись получает стартовую стадию `Скоринг`.
- Если ingest распознал данные, они автоматически появляются в соответствующих полях.
- Если ingest не распознал данные, соответствующие поля показываются как disabled/read-only, а не как активные пустые controls.
- Пользователь может визуально отличить заполненное поле от незаполненного и выключенного.
- Существующий create flow на `month` и `detail` не расходится по поведению с `home`.

### Functional Test Pack

`FT-FS-001` Open create popup from home
- Открыть `/`.
- Нажать `Добавить проект`.
- Expect: открывается popup создания проекта.

`FT-FS-002` Validation of required inputs
- Открыть popup на главной.
- Не заполнять название или не выбирать архив.
- Expect: подтверждение недоступно или запрос не отправляется.

`FT-FS-003` Create project from archive
- На главной открыть popup.
- Ввести название проекта.
- Выбрать архив.
- Подтвердить создание.
- Expect: backend создает запись и интерфейс переводит пользователя на detail новой записи.

`FT-FS-004` Default stage is scoring
- После создания новой записи открыть ее detail.
- Expect: стартовая стадия записи — `Скоринг`.

`FT-FS-005` Recognized data is populated
- Использовать архив, в котором есть распознаваемые данные.
- Expect: соответствующие поля detail заполнены автоматически после ingest.

`FT-FS-006` Missing data is visually disabled
- Проверить поля, по которым ingest ничего не вернул.
- Expect: они отображаются выключенными или read-only placeholders.
- Expect: они не выглядят как обычные активные пустые поля, готовые к ручному вводу без явного действия пользователя.

`FT-FS-007` User can distinguish missing vs filled data
- Сравнить заполненные и незаполненные после ingest поля.
- Expect: визуальное различие между `значение пришло` и `данных нет` очевидно без дополнительного объяснения.

`FT-FS-008` Header create action stays consistent
- Повторить открытие popup через `Добавить проект` на `home`, `month`, `detail`.
- Expect: popup, валидация и поведение создания совпадают на всех трех страницах.

### Risks

- Нужно отдельно договориться, является ли disabled-state только визуальным состоянием после ingest или строгим запретом на ручное редактирование до отдельного действия пользователя.
- Если backend сейчас создает запись раньше завершения ingest, понадобится явное промежуточное состояние `данные загружаются`.
- История затрагивает не только UI popup, но и продуктовый контракт между archive upload, ingest и detail rendering.
