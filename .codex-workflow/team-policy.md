# Team Policy

Этот файл фиксирует постоянную команду и модель прав для `scoring/`.

## Standing Team

Базовый состав:

- `Manager`
- `Primary Developer`
- `Primary QA`

Временные роли:

- `Test Automation`
- `Design Agent`

Расширенная команда по умолчанию не нужна.
Лишние агенты не должны жить между циклами без явной причины.

## Codex Worker Rule

- Если manager поднимает implementation-агента, developer-worker по умолчанию запускается на Codex-модели.
- Базовая модель для developer-pass в текущем контуре: `gpt-5.3-codex`.
- Отклоняться от этого правила можно только по явной причине:
  - пользователь прямо попросил другую модель;
  - задача требует другого режима по техническому ограничению среды;
  - manager явно фиксирует override в board или handoff.
- Это правило относится именно к implementation-worker'ам.
- Manager и QA могут оставаться в основной консоли, если отдельный агент для них не нужен.

## Role Duties

### Manager

Обязанности:

- держать цель текущего цикла;
- поддерживать board;
- назначать implementation scope;
- назначать сценарии проверки;
- делать финальный короткий smoke-check;
- принимать или отклонять результат;
- закрывать временных помощников.

Права:

- широкий read по всему проекту;
- write к workflow-документам и узким интеграционным правкам.

Почему так:

- manager нужен полный контекст;
- manager должен координировать, а не автоматически становиться основным исполнителем всего проекта.

### Primary Developer

Обязанности:

- реализовать назначенный bounded task;
- удерживать изменения внутри делегированной поверхности;
- выполнить честную локальную проверку;
- вернуть компактный handoff и остаточные риски.

Права:

- write только к назначенной поверхности реализации.

Почему так:

- этого достаточно, чтобы довести задачу до результата;
- ограничение scope снижает шум, риск регрессий и расползание цикла.

### Primary QA

Обязанности:

- проверить измененную поверхность;
- прогнать назначенные manager-сценарии;
- проверить фактическое поведение и доступность артефактов;
- отдать findings first.

Права:

- read-only по умолчанию;
- разрешены только non-mutating checks.

Почему так:

- QA должен оставаться независимым приемочным слоем;
- read-only режим не позволяет "починить" проблему незаметной правкой.

### Test Automation

Обязанности:

- добавлять или восстанавливать автоматические проверки;
- закрывать регрессии на хрупких участках.

Права:

- write только при явном назначении;
- по возможности test-only scope.

Почему так:

- автоматизация иногда требует кода;
- но эта роль не должна бесконтрольно расширяться в продуктовую реализацию.

### Design Agent

Обязанности:

- проверять качество внутренних UI-поверхностей;
- давать узкие рекомендации по иерархии, плотности, подаче и читаемости;
- выполнять bounded design-first pass, если manager явно его назначил.

Права:

- read-only по умолчанию;
- узкий write только для явно назначенной UI-поверхности.

Почему так:

- визуальный цикл требует отдельной петли обратной связи;
- это не должен быть скрытый канал для изменения бизнес-логики.

## Scenario Rule

Любой цикл, который затрагивает:

- статическую страницу,
- workflow-контур,
- runtime-скрипты,
- генерацию coding-артефактов,

должен получить точные сценарии для QA.

Manager должен ссылаться на:

- [test-scenarios.md](/C:/Users/illki/Desktop/projects/scoring/.codex-workflow/test-scenarios.md)

QA должен всегда возвращать:

- scenarios run
- scenarios passed
- scenarios failed
- scenarios not run
- residual risks

## Manager Final Smoke Check

После developer и QA pass manager должен сделать короткий финальный check.

Формат:

- `works`
- `does not work`

Примеры:

- `agent contour docs: works`
- `scoring page update: works`

## Escalation Rule

Если цикл касается:

- схем данных;
- Python-скриптов в `mvp/scripts`;
- экспортных артефактов;
- пользовательской страницы или будущего web UI,

имеет смысл рассматривать отдельный `Test Automation` или `Design Agent` pass.

## Cleanup Rule

По окончании цикла:

- закрывать временных помощников;
- не держать idle developer/tester/design agents без немедленного следующего шага;
- переносить вперед только принятые решения и workflow-артефакты.
