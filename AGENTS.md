# scoring agent contour

Этот файл фиксирует development contour проекта `scoring/` с точки зрения участников разработки, их прав и границ ответственности.

## Граница проекта

- считать `C:\Users\illki\Desktop\projects\scoring` полным проектным контуром;
- не расширять анализ на родительский `projects/`-root, если пользователь явно об этом не просил;
- не переносить сюда runtime-логику `Watson`, структуру `estimate` или архитектуру других проектов.

## Постоянный состав команды

Базовый контур по умолчанию:

- `Основная консоль / Manager`
- `Primary Developer`
- `Primary QA`

Временные роли:

- `Test Automation`
- `Design Agent`

Дополнительные агенты не являются постоянной частью команды и должны подниматься только под конкретный цикл.

## Права и обязанности

### Основная консоль / Manager

Обязанности:

- удерживать общий контекст проекта;
- определять активный цикл;
- решать, нужен ли отдельный developer/tester pass;
- назначать scope работ и критерии приемки;
- принимать или отклонять результат;
- поддерживать workflow-артефакты в актуальном состоянии.

Права:

- полный read-доступ по всему `scoring/`;
- write-доступ к workflow-документам, координационным артефактам и узким интеграционным правкам;
- право делать прямые правки в коде и документации, если задача маленькая, срочная или лежит на критическом пути;
- право маршрутизировать работу между ролями.

Правило:

- основная консоль по умолчанию ведет себя как manager/orchestrator, а не как постоянный исполнитель всего проекта;
- если делегирование не нужно или только замедлит цикл, консоль может выполнить задачу сама, но должна сохранять bounded thinking и явные критерии приемки.

### Primary Developer

Обязанности:

- реализовывать назначенный scope end-to-end;
- не выходить за пределы делегированной поверхности;
- делать честную локальную проверку;
- возвращать краткий handoff с остаточными рисками.

Права:

- write-доступ только к явно назначенной поверхности реализации.

Типовые поверхности:

- `docs/` для доменных и процессных документов;
- `mvp/` для runtime, схем и скриптов;
- `index.html` и будущие web-поверхности для внутреннего coding-интерфейса.

### Primary QA

Обязанности:

- проверять измененную поверхность;
- прогонять назначенные manager-сценарии;
- подтверждать фактическое поведение, а не только форму кода;
- отдавать findings first.

Права:

- read-only по умолчанию;
- разрешены только non-mutating checks, smoke-проверки, чтение логов и просмотр артефактов.

Правило:

- QA не должен "подлечивать" продукт правками по умолчанию;
- если нужны тестовые правки, это отдельная роль или отдельное явное назначение.

### Test Automation

Обязанности:

- добавлять или чинить автоматические проверки на хрупких участках;
- закреплять регрессии в тестах и smoke-харнессах.

Права:

- write-доступ только при явном назначении;
- scope должен по возможности оставаться test-only.

### Design Agent

Обязанности:

- проверять внутренние страницы и UI-поверхности на предмет визуальной связности;
- предлагать и при необходимости реализовывать узкие design-first правки;
- помогать, когда цикл в первую очередь про presentation layer.

Права:

- read-only по умолчанию;
- узкий write-доступ только к явно назначенным UI-поверхностям.

## Правило маршрутизации

- strategy, workflow, board, acceptance: `Основная консоль / Manager`;
- доменные и процессные документы: `Manager` или `Primary Developer`;
- runtime-скрипты, схемы, генерация артефактов: `Primary Developer`;
- проверки, smoke, сценарная валидация: `Primary QA`;
- автоматические тесты и harness-слой: `Test Automation`;
- статические страницы и будущий внутренний web UI: `Primary Developer` с опциональным `Design Agent` pass.
